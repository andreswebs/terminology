# E2.T11 — Dialect registration (linguist/register.go)

## Goal

Wire the Linguist reader and writer into the dialect registry via init-time registration. This breaks the import cycle between the tbx package (which owns the registry) and the linguist package (which provides the concrete implementations).

## Refs

- E2 spec: [docs/specs/002-domain-and-io.md](docs/specs/002-domain-and-io.md) §"Architecture" — \`linguist/register.go\` in the file layout
- E2 spec: §"Domain types" — "Dialect identifier lives as \`tbx.Dialect\` rather than as a hardcoded branch, so a future TBX-Basic reader/writer is a registered implementation"

## Files to create

- \`src/internal/tbx/linguist/register.go\`

No separate test file needed — registration is tested by T12 (Load dispatches through it) and T14 (round-trip tests import the linguist package).

## Implementation

```go
package linguist

import "github.com/andreswebs/terminology/internal/tbx"

func init() {
    tbx.RegisterDialect(
        tbx.DialectLinguist,
        func() tbx.Reader { return NewReader() },
        func() tbx.Writer { return NewWriter() },
    )
}
```

Design notes:
- **init() registration** is the standard Go pattern for plugin-style dispatch. The linguist package registers itself when imported.
- **Factory functions** (\`func() tbx.Reader\`) rather than singletons ensure each Load/Save gets a fresh instance.
- **Import cycle prevention**: \`tbx\` defines the interfaces and registry. \`linguist\` imports \`tbx\` and registers. Consumers import \`linguist\` with a blank import (\`_ "...linguist"\`) to trigger registration. This is the same pattern used by \`database/sql\` drivers, \`image\` decoders, etc.
- **Consuming code must blank-import**: any test or command that calls \`tbx.Load\` must have \`_ "github.com/andreswebs/terminology/internal/tbx/linguist"\` in its imports, or Load will fail with ErrUnsupportedDialect because no reader is registered.
